Windows Forensics Analysis PT III (Memory)

A Step-by-Step Walkthrough: Analyzing a SolarMarker-Infected Memory Dump

Disclaimer

This guide is purely for educational and personal reference. Every technique, plugin, and insight comes directly from the SANS GCFA training I just finished. I’m walking through a real SolarMarker memory sample (solarmarker.img) so you can follow along, understand the “why” behind every step, and see how the pieces fit together. All credit to SANS — their training is phenomenal.

What We’re Looking At

We have a memory image from a Windows 10 machine that got hit by SolarMarker (also called Jupyter or Yellow Cockatoo).

This malware is sneaky: it’s a .NET infostealer that grabs autofill data, saved passwords, and credit cards from browsers, plus it can download more files and run commands from its C2 server.

The infection chain starts with a big fake “flash_installer.msi”, drops PowerShell scripts, creates a weird random folder full of encrypted junk, and plants a .lnk in the Startup folder for persistence.

Great background reads:Cybereason Solarmarker article, Palo Alto Unit 42 SolarMarker article

Tools I Used

  • Volatility 3 (running on SIFT Linux) — super granular, perfect for digging into specific structures.
  • Volatility 2(a couple plugins still work better here).
  • MemProcFS (on Windows SIFT) — my favorite for this case because it mounts the entire memory dump as a normal drive (M:) and gives you every artifact in one folder structure. Run it once and you’re done.

Why Memory Analysis Matters

Everything that happens on a computer touches memory: running processes, threads, network connections, open files, passwords, encryption keys, registry values — even things that were deleted from disk. If you only look at the hard drive, you miss the full story. Memory shows you what was actually executing right when the snapshot was taken.

The Standard Analysis Flow We’ll Follow

  1. Profile the system (know what OS and kernel we’re dealing with)
  2. Find rogue processes
  3. Dig into those processes objects (DLLs, command lines, tokens, handles)
  4. Check network activity
  5. Look for code injection / anomalies
  6. Audit drivers / Rootkit detection
  7. Dump anything suspicious

Quick Windows Memory 101 (Why This Stuff Matters)

Think of memory like a big linked list:

  • KDBG is the starting point — the kernel debugger block that points to every active process.
  • EPROCESS blocks are chained together like a linked list. Each block represents one running process (PID, creation time, threads, etc.).
  • VAD Tree tells you what’s actually loaded in that process’s memory (code sections, heap, mapped files).
  • PEB is the process’s own “ID card” — it lists loaded DLLs and the exact command line that was used to start it.
  • Handles show every file, registry key, or socket the process is touching.
  • Access Tokens tell you who the process is running as (SYSTEM vs. domain user).
  • Threads are the actual lines of code executing right now.

Analyst mental model I always use:

EPROCESS
├── PEB     → “What the process claims is loaded”
├── VAD     → “What is really in memory”
├── Handles → “What it’s touching”
├── Token   → “Who it’s running as”
└── Threads → “What is actually executing”

Identity  → EPROCESS
Intent    → Command line (PEB)
Execution → Threads + VAD
Access    → Handles
Authority → Token

Analysis Steps

Step 1: Profile the System

Why this matters

Profiling the system is the very first step because it tells Volatility exactly how to interpret the memory image. Memory layouts, kernel structures, and offsets can vary between Windows versions and builds. Without the correct profile, all subsequent analysis could be inaccurate or even completely misleading.

Additionally, taking note of the system timestamp and other environment details helps us begin building a timeline of activity. This context is crucial when correlating suspicious processes, network connections, or artifacts to specific points in time during an investigation.

In short, profiling ensures that the tools know what “normal” looks like for this system, making it much easier to spot anomalies in later steps.

I always start here so Volatility knows exactly which symbols and offsets to use. 

# Step 1: Check the memory image info
vol -f solarmarker.img windows.info

# Step 2: Based on output, set profile (Volatility 2 example)
vol -f solarmarker.img --profile=Win10x64_19041 pslist

We get: Windows 10 x64, build details, Kernel Base address, DTB, system time (2022-06-01 09:39:50 UTC), etc. Everything lines up — good to go.

A good notes as well to take away from this output is the timestamp of the memory sample. This will help us start to build the foundation of our timeline.

Step 2: Identify Rogue Processes

Why this matters

Identifying rogue processes is a critical first step in memory analysis because processes represent the active execution layer of an attack. By analyzing what is running (and what may have recently run), we can begin to uncover how an attacker gained execution, escalated privileges, and moved within the system.

Many attack techniques—such as fileless malware, living-off-the-land binaries (LOLBins), and process injection—rely on abusing legitimate processes to blend in with normal system activity. This makes it essential to closely examine process relationships, execution context, and anomalies in behavior rather than relying solely on known malicious filenames.

Let’s begin by examining the list of currently running processes. We will use two Volatility plugins for this step: pslist and psscan.

windows.pslist (running processes) + windows.psscan (scans for terminated/hidden processes) + Volatility 2 pstree for the tree view.

When reviewing the results, look for anomalies such as:

  • Unusual or unexpected process names
  • Processes running from suspicious locations
  • Duplicate or misspelled system process names
  • Processes with abnormal parent-child relationships
  • Recently terminated processes that appear suspicious

IOCs noted:

  • services.exe → svchost.exe → msiexec.exe (PID 5340, SYSTEM)
  • msiexec (5340) → msiexec.exe (PID 6872, now domain user)
  • msiexec (6872) → powershell.exe (PID 5352) + conhost.exe (PID 6388)

Classic red flag: msiexec spawning PowerShell + privilege context switch.

Step 3: Analyze Process Objects

In this step, we aim to build context around previously identified Indicators of Compromise (IOCs) while also looking for potential signs of additional IOCs.

To do this, we will examine several key details:

  • Associated DLLs:Identify dynamic link libraries loaded by the process, as these can reveal malicious behavior or relationships with other compromised components.
  • Command Line Arguments:Review the command-line arguments used by the process to uncover potentially suspicious flags, payloads, or obfuscation techniques.
  • Security Content:Assess what security content (e.g., antivirus, firewall, or endpoint protection tools) the process is interacting with, which could provide insights into the attack’s evasion tactics.
  • Handles/Objects Accessed:Look into the handles or objects the process is interacting with (files, registry keys, network connections, etc.), as these may provide clues about the process’s intentions and any malicious activity.

The first thing we’ll examine is windows.cmdline, as this is relevant to the IOC we’re investigating. This particular IOC triggered a PowerShell session that, in turn, spawned a conhost process. By analyzing the command line arguments associated with this process, we can gain further insight into how the PowerShell session was executed and whether any suspicious or malicious flags were used.

Unfortunately, there were no command-line arguments observed for the PowerShell or conhost processes (via cmdline in Volatility 2), nor did we see any relevant output from the dlllist plugin.

Next, we need to examine the security context in which the PowerShell and conhost processes were running. Both processes are running as a domain user, which, depending on the user, may not be out of the ordinary. However, what stands out is the parent process—it appears that powershell and conhost were both spawned by msiexec.exe and subsequently by services.exe. This raises some questions: Why was msiexec.exe involved in launching these processes, and what triggered it to do so, especially when considering the normal behavior of these processes?

Step 4: Review Network Objects

Here, we have two main plugins for examining network connections. Netstat displays all currently allocated and active connections that are still present. Netscan, on the other hand, scans memory to identify remnants of ended or unallocated connections

By using both tools together, we can build a clearer picture of whether processes were communicating with external systems. This helps us identify suspicious external IPs, unusual ports, and processes that should not be communicating externally but stand out.

When reviewing the results, look for anomalies such as:

  • Unusual or unexpected external IP addresses (especially rare, foreign, or known malicious ranges)
  • Connections to suspicious or uncommon ports (e.g., high-numbered ports, non-standard service ports)
  • Processes that should not normally communicate over the network (e.g., lsass.exe, winlogon.exe)
  • Multiple outbound connections from a single process to different IPs (possible C2 beaconing or scanning behavior)
  • Repeated connections to the same external IP (potential command-and-control communication)
  • Short-lived or rapidly opening/closing connections (beaconing, failed callbacks, or scanning activity)
  • Connections using uncommon protocols or address types (e.g., unexpected IPv6 usage in an IPv4-dominant environment)
  • Network activity from previously identified suspicious processes or IOCs
  • Connections occurring under unusual user contexts (e.g., SYSTEM processes making external connections unexpectedly)
  • Listening ports opened by suspicious or unknown processes

Here, we see the output from Netscan. There are no connections tied to our previous IOC chain. However, we do observe three external closed connections, all using IPv6. Only one of these has a single user report of suspicious activity, but none have enough evidence to be labeled as having a bad reputation. Aside from that nothing in particular stands out for network connections on this host.

Step 5: Look for code injection

What is Code Injection? At its core code injection aims to execute malcious code in a target "victim" process. Some examples of that are below.

Common Code Injection Types:

  • Simple DLL Injection Write a DLL path into remote process → force it to api call LoadLibrary(path) via remote thread. (Easiest, but leaves obvious artifacts such as (disk file + loaded module list).)
  • Reflective DLL Injection Write entire DLL binary into remote process → remote thread jumps to custom ReflectiveLoader export → DLL manually maps/fixes/loads itself (no LoadLibrary needed). (Popular because it's fileless and skips standard loader APIs.)
  • Process Hollowing Create legit process suspended → gut/hollow its original code → write full malicious EXE → fix headers/PEB/context → resume thread. (Makes malicious code appear as if it's the legitimate program (great for blending in).)

Here, we focus on two powerful Volatility plugins for hunting code injection: windows.ldrmodules and windows.malfind.

windows.ldrmodules compares the three PEB loader lists (InLoad, InMemory, InInit) to detect unlinked/hidden DLLs — malware often removes itself from one or more of these lists to evade dlllist while still remaining loaded in memory. Look for discrepancies where a module appears in one list but not others; that's a classic sign of stealthy DLL injection.

windows.malfind is often the golden ticket for spotting injected code. It scans process memory for suspicious regions that are:

  • Marked executable (e.g., PAGE_EXECUTE_READWRITE — RWX is almost always suspicious in user-mode processes)
  • Not backed by a file on disk (private memory, no mapped EXE/DLL)
  • Often containing MZ/PE headers, shellcode patterns, or anomalous VAD tags

Malfind highlights potential injected shellcode, reflective DLLs, or manually mapped PE files that standard module enumeration misses. Use the --dump-dir option to extract suspicious memory sections for YARA scanning, strings analysis, or reverse engineering — many analysts consider this the single most valuable step for confirming in-memory-only malware.

Together, these plugins catch both hidden DLLs (ldrmodules) and generic injected code/shellcode (malfind), giving you broad coverage against common injection techniques like reflective loading, process hollowing artifacts, and classic code caves.

When reviewing the results, look for anomalies such as:

  • Processes with memory regions marked as RWX (Read-Write-Execute), which may indicate injected or shellcode regions
  • Presence of injected DLLs that do not exist on disk or are not linked through normal module loading mechanisms
  • Processes with abnormal or missing entries in their loaded module lists (e.g., unlinked or hidden modules)
  • Threads starting at suspicious memory addresses (e.g., not within known module boundaries)
  • Evidence of API patterns associated with injection (e.g., VirtualAllocEx, WriteProcessMemory, CreateRemoteThread)
  • Processes whose in-memory image does not match the expected executable on disk (possible process hollowing)
  • Suspicious parent-child relationships combined with memory anomalies (e.g., legitimate process spawning unusual child with injected code)
  • Indicators of manual PE loading (e.g., missing headers, improper section alignment, or absence of LoadLibrary usage)
  • Processes with unusually high memory entropy (potential packed or encrypted payloads)
  • Signs of tampering with the Process Environment Block (PEB) or thread context

Unfortunately for us Volatility did not show any results for either module. It happens and the reason why we use mutiple tools and techniques.

Step 6: Audit Drivers

The next step in our memory analysis process is to focus on the drivers loaded into the system. Drivers are critical system components that run in kernel mode and can be exploited by attackers to maintain persistence, escalate privileges, or even execute arbitrary code. By auditing the drivers present in memory, we can identify any suspicious or rogue drivers that may have been loaded by malicious actors.

Driver manipulation is a common tactic for rootkits, fileless malware, and other advanced threats, as they operate at the kernel level and can evade detection by traditional endpoint protection tools. Therefore, we need to ensure we are examining both active drivers and any potentially hidden or unloaded drivers that may be lurking in memory.

Key Artifacts to Review:

  • Driver Names and File Paths: Suspicious or unusual driver names can indicate the presence of malware. Attackers often hide malicious drivers with names resembling legitimate system files, or use random, obfuscated names to evade detection.
  • Driver Memory Locations: Checking the memory addresses of loaded drivers is crucial. A driver loaded into unexpected memory ranges could be indicative of manipulation or injection.
  • Driver Signed Status: Legitimate drivers are typically signed with a valid certificate. Drivers that are unsigned or have invalid signatures may be malicious or tampered with.
  • Process Creation from Drivers: Drivers are often used to spawn malicious processes or maintain control over the system. Investigating any processes that were created by drivers can uncover covert attacks.

In this step, we’ll use Volatility plugins to extract driver information and analyze it for any signs of malicious activity.

Volatility Plugins for Rootkit & Driver Auditing

The following plugins help detect kernel-level anomalies, hidden drivers, hooked tables, and rootkit techniques. Compare Volatility 2 and 3 equivalents for cross-version reference.

Volatility 2 Plugin Volatility 3 Plugin Description
apihooks N/A Find userland IAT, inline, and trampoline hooks (user-mode API hooking often used by rootkits/malware).
idt N/A Display Interrupt Descriptor Table entries (check for IDT hooks that redirect interrupts).
ssdt windows.ssdt Display System Service Descriptor Table entries (detect SSDT hooks that alter kernel system calls — a classic rootkit technique).
driverirp windows.driverirp Identify I/O Request Packets (IRP) hooks (malicious drivers often hook IRPs to intercept file/network/registry operations).
psxview windows.psxview Find hidden processes via cross-view techniques (compares multiple process enumeration methods to spot DKOM/rootkit-hidden processes; can indirectly reveal driver-based hiding).
modules windows.modules View list of loaded kernel drivers/modules (baseline for detecting anomalies or mismatches).
modscan windows.modscan Find drivers via pool tag scanning (uncovers hidden/unlinked drivers that evade normal module lists — key for rootkit detection).
N/A windows.drivermodule Find loaded drivers hidden by a rootkit (specialized cross-view check for stealthy kernel modules).

Sample Commands (Volatility 3)

# Check SSDT for hooks (kernel system call redirection)
vol -f solarmarker.img windows.ssdt

# Identify IRP hooks in drivers
vol -f solarmarker.img windows.driverirp

# Cross-view hidden processes (helps spot driver/rootkit hiding)
vol -f solarmarker.img windows.psxview

# List loaded kernel modules/drivers
vol -f solarmarker.img windows.modules

# Scan for hidden/unlinked drivers via pool tags
vol -f solarmarker.img windows.modscan

# Detect drivers specifically hidden by rootkit techniques
vol -f solarmarker.img windows.drivermodule

Run these plugins if you suspect kernel-level manipulation (e.g., after spotting anomalies in processes, network, or memory regions). Look for mismatches between list vs. scan results, unusual hooks, or drivers not backed by expected files.

By running these commands, we gather the list of drivers loaded into memory, including their names, memory addresses, and other critical metadata. For example, if we notice a driver loaded at an unusual address or a driver with an odd name that doesn’t match typical Windows drivers, it could be a potential indicator of compromise (IOC).

Red Flags to Look For:

  • Drivers with suspicious or obfuscated names (e.g., random alphanumeric strings or names resembling legitimate system drivers with small variations).
  • Drivers loaded from unusual locations (e.g., non-system directories like `C:\Windows\Temp` or `C:\Users\Public\`).
  • Unsigned drivers or drivers with invalid signatures.
  • Drivers that spawn suspicious processes or exhibit abnormal behavior (e.g., they initiate outbound network connections or manipulate system settings).
  • Presence of hidden drivers detected by tools such as modscan or drivermodule.
  • I/O request packet (IRP) hooks that indicate possible rootkit activity.

If any of these red flags are found, further investigation should be conducted to analyze the potential malicious driver’s behavior. You may need to extract the driver and perform static or dynamic analysis to understand its full capabilities and intent.

Step 7: Dump Suspicious Artifacts

Once we’ve identified suspicious artifacts, the next step is to extract them from the larger memory sample and perform in-depth analysis on each item. This can include techniques such as static code analysis to better understand the artifact’s behavior and intent.

There are four main types of artifacts we can extract:

  • Process Memory Sections: Useful for examining the raw data in memory, including hex and ASCII representations of what was present at runtime.
  • Image-Mapped Files (DLL/EXE/SYS): Critical for determining code intent and identifying malicious indicators of compromise (IOCs) within binaries.
  • Cached File Objects: These may include files or objects accessed by a malicious process. For example, a PowerShell process may reference a file containing commands or arguments for execution.
  • Specialized Objects (MFT, Registry Hives): Useful for correlating file activity through MFT entries or extracting registry data when investigating persistence mechanisms, such as obfuscated PowerShell stored in registry keys.

MemProcFS — The All-in-One View

Moving on to MemProcFS, this tool works differently from Volatility. Instead of running individual plugins to examine memory piece by piece, MemProcFS processes the entire memory image and effectively runs multiple plugins at once. It mounts the results to a virtual file system (typically the M: drive), allowing you to explore memory artifacts using a familiar Windows folder structure. You’ll see output similar to Volatility plugins but in a single, unified view. This method is advantageous because you can run it once and have everything you need to start your investigation.
